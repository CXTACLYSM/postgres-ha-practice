# postgres-ha-practice

> Воспроизводим connection exhaustion в PostgreSQL под нагрузкой и решаем проблему через streaming replication и HAProxy. Каждый шаг наблюдаем в реальном времени через Prometheus + Grafana. Go + Docker Compose lab.

---

## Зачем этот репозиторий

Это практический эксперимент основанный на реальной production-проблеме. При определённом сочетании параметров — размер пула соединений, уровень конкурентности и время выполнения запроса — pgx connection pool исчерпывается, горутины встают в очередь, и запросы начинают падать по таймауту. Не потому что запросы плохо написаны. Не потому что неправильно настроен PostgreSQL. А потому что один узел физически не может обслуживать столько одновременных соединений.

Репозиторий структурирован как два последовательных эксперимента. В первом вы воспроизводите проблему на одном узле и наблюдаете в Grafana как пул упирается в потолок и начинают расти таймауты. Во втором — добавляете реплики за HAProxy и без единого изменения в коде приложения видите как те же графики возвращаются в норму.

## Математика проблемы

### Закон Литтла

В 1961 году математик Джон Литтл доказал теорему которая стала фундаментом теории массового обслуживания. В её самой общей формулировке она звучит так: среднее число объектов в стабильной системе равно среднему темпу их поступления умноженному на среднее время которое объект проводит в системе. Для нашей системы где «объект» — это HTTP-запрос, «система» — это pgx connection pool, «темп поступления» — это RPS, а «время в системе» — это время выполнения SQL-запроса, закон принимает конкретную форму:

```
required_connections = concurrency × query_duration_seconds
```

Это не приближение и не эмпирическое правило. Это математически доказанное равенство которое выполняется для любой стабильной очереди при любой нагрузке. Именно поэтому мы можем не просто наблюдать поведение системы на графиках, но и **предсказывать** его до запуска эксперимента.

### Правильная интерпретация формулы

Формула `required_connections = concurrency × query_duration` описывает **установившееся состояние** — то есть сколько соединений одновременно занято при стабильном потоке запросов. При максимальной пропускной способности пула (`250 req/s`) и `query_duration = 0.1s` в системе одновременно занято ровно `250 × 0.1 = 25` соединений — в точности `MaxConns`. Это и есть точка насыщения: пул работает на полную мощность без очереди.

Ошибочно применять эту формулу как `required_connections = concurrency × query_duration = 50 × 0.1 = 5` и делать вывод «всё в порядке». `concurrency` и `RPS` — это разные величины. `concurrency = 50` означает что 50 горутин **одновременно** давят на сервер, а не что система обрабатывает 50 запросов в секунду. При насыщённом пуле реальный RPS определяется пропускной способностью пула (`MaxConns / query_duration = 25 / 0.1 = 250 req/s`), а не уровнем concurrency. Формулу Литтла нужно применять к пулу, а не ко входящей нагрузке: пул обрабатывает `250 req/s` и держит `25` соединений занятыми — это согласованно. Остаток concurrency (`50 - 25 = 25` горутин) стоит в очереди.

### Детальная декомпозиция: MaxConns=25, concurrency=50, query_duration=100ms, context_timeout=150ms

Разберём полный жизненный цикл одного цикла нагрузки по строкам кода.

**t = 0ms.** Apache Bench держит 50 HTTP-соединений открытыми. Для каждого запроса chi запускает горутину которая проходит цепочку middleware. `Metrics.Handle` фиксирует `start := time.Now()` и вызывает `next.ServeHTTP(rec, r)` — с этого момента middleware **ждёт** и не продвигается дальше. `TargetHandler.ServeHTTP` создаёт контекст с дедлайном `150ms` — таймер начинает тикать немедленно, независимо от того получит горутина соединение или будет стоять в очереди. Затем вызывается `h.Slow.Handle(ctx, ...)` и внутри него `h.pool.Exec(ctx, "SELECT pg_sleep(0.100)")`. Внутри `pool.Exec` первой операцией идёт `pool.Acquire(ctx)` — получить соединение из пула.

**t = 0ms, разветвление судьбы.** Из 50 горутин которые одновременно вызвали `pool.Acquire` — первые 25 получают соединение немедленно и начинают выполнять `SELECT pg_sleep(0.100)` в PostgreSQL. Оставшиеся 25 горутин не получают соединение — пул исчерпан. Они паркуются планировщиком Go прямо на строке `pool.Acquire(ctx)` и встают во внутреннюю очередь pgxpool. Горутины живы, занимают память, но не потребляют CPU — просто ждут пока кто-то вернёт соединение.

**t = 100ms.** PostgreSQL завершает `pg_sleep(0.100)` для первых 25 горутин. Каждая возвращает соединение в пул. pgxpool видит 25 ожидающих и немедленно будит их — каждая получает соединение и начинает выполнять `SELECT pg_sleep(0.100)`. Но у этих горутин уже прошло 100ms из 150ms отведённого контекста. Им нужно ещё 100ms для выполнения запроса — а осталось только 50ms.

**t = 150ms.** Контекст истекает для всех 25 горутин которые начали выполнение в t=100ms. Go отменяет контекст. pgxpool замечает отмену и прерывает `pool.Exec` — возвращает ошибку `context deadline exceeded`. Каждая из этих горутин получает `err != nil`, выполняет `w.WriteHeader(http.StatusServiceUnavailable)` — `statusRecorder` перехватывает вызов и записывает `r.status = 503`. Хэндлер возвращается. Управление передаётся обратно в `Metrics.Handle` который вычисляет `elapsed`, проверяет `rec.status == 503` и вызывает `m.prometheus.HttpRequestsTotal.WithLabelValues("timeout").Inc()`. Prometheus при следующем scrape заберёт это значение и Grafana покажет рост красного ряда.

**Итог одного цикла.** Из 50 горутин ровно 25 всегда возвращают `success` (`0ms ожидания + 100ms выполнения = 100ms < 150ms`), ровно 25 всегда возвращают `timeout` (`100ms ожидания + 100ms выполнения = 200ms > 150ms`). Это не вероятность и не случайность — это детерминированный результат который повторяется при каждом цикле пока параметры не изменятся. RPS системы при этом ровно 250 — именно столько запросов в секунду успевает пройти через пул. Вторые 25 горутин в каждом цикле не добавляют полезную работу, а только создают нагрузку на очередь и генерируют таймауты.

### Теорема о гарантированном таймауте

Обобщим наблюдение в точную формулировку. Запрос в хвосте очереди гарантированно получит таймаут если выполняется неравенство `wait_time + query_duration > context_timeout`, где `wait_time = (queue_depth / MaxConns) × query_duration` и `queue_depth = concurrency - MaxConns`. Подставляя: `((concurrency - MaxConns) / MaxConns) × query_duration + query_duration > context_timeout`. Упрощая: `query_duration × (concurrency / MaxConns) > context_timeout`.

Для наших параметров: `0.1s × (50 / 25) = 0.2s = 200ms > 150ms` — теорема выполняется, таймаут гарантирован. Это детерминированное следствие из трёх чисел, а не вероятностное утверждение.

### Горизонтальное решение

Из той же математики следует почему вертикальное масштабирование не работает. Увеличить `MaxConns` — значит переложить нагрузку на PostgreSQL, у которого модель process-per-connection: каждое соединение порождает отдельный OS-процесс с overhead ~5–10MB памяти. При 500 соединениях это уже 2.5–5GB только на overhead до начала какой-либо работы. Физический предел достигается раньше чем математический.

Горизонтальное масштабирование через `r` реплик меняет формулу принципиально: эффективный пул становится `MaxConns × r`, порог насыщения сдвигается в `r` раз, и то же условие гарантированного таймаута перестаёт выполняться при той же нагрузке. При `r = 2` и тех же параметрах: `wait_time = (queue_depth / (MaxConns × 2)) × query_duration = (25 / 50) × 100ms = 50ms`. Суммарно `50ms + 100ms = 150ms` — ровно на границе контекста. Добавление третьей реплики даёт запас: `(25 / 75) × 100ms + 100ms ≈ 133ms`. Все запросы успевают с margin.

## Три параметра эксперимента

Система управляется тремя независимыми параметрами. В каждой серии экспериментов два из них фиксируются, третий меняется — это единственный способ честно установить причинно-следственную связь.

**Параметр 1 — concurrency** (`-c` в Apache Bench). Количество горутин которые одновременно давят на сервис. Определяет глубину очереди к пулу: `queue_depth = concurrency - MaxConns`.

**Параметр 2 — query_duration** (`pg_sleep` в SQL). Время которое одно соединение удерживается занятым. Вместе с concurrency определяет `required_connections` по закону Литтла.

**Параметр 3 — context timeout** (`context.WithTimeout` в хэндлере). Максимальное время которое HTTP-запрос готов ждать полного выполнения включая ожидание соединения из пула. Определяет границу между `success` и `timeout` на графике Grafana.

## Архитектура

```
                        ┌─────────────────┐
                        │   Apache Bench  │
                        │   (load tank)   │
                        └────────┬────────┘
                                 │ HTTP
                        ┌────────▼────────┐
                        │   Go Service    │
                        │   (chi router)  │
                        │                 │
                        │  WritePool ─────────► PostgreSQL Primary :5432
                        │  ReadPool  ─────────► HAProxy :5433
                        └────────┬────────┘          │
                                 │ scrape        ┌───▼─────────┐
                        ┌────────▼────────┐      │  Replica 1  │
                        │   Prometheus    │      │  Replica 2  │
                        └────────┬────────┘      └─────────────┘
                                 │
                        ┌────────▼────────┐
                        │     Grafana     │
                        └─────────────────┘
```

Go-сервис держит два независимых pgx connection pool инжектированных через dependency injection. **WritePool** смотрит напрямую на primary и обслуживает операции записи (Command-сторона в терминах CQRS). **ReadPool** смотрит на HAProxy, который прозрачно распределяет читающие запросы по репликам. Приложение ничего не знает о топологии репликации — оно знает только один read-эндпоинт.

Все горутины которые выполняют читающие запросы работают с одним и тем же объектом `*pgxpool.Pool` через указатель, переданный при инициализации. Никаких копий — один пул, много потребителей, внутренняя синхронизация через атомарные операции обеспечивает безопасность конкурентного доступа.

## Метрики и Grafana

В стандартной библиотеке Go нет встроенных HTTP-метрик — `net/http` ничего не экспортирует. Все метрики регистрируются через `prometheus/client_golang` и отдаются на `GET /metrics` в Prometheus exposition format.

**RPS** считается через `Counter` — монотонно растущий счётчик с лейблом `status: success` или `status: timeout`. Grafana вычисляет RPS через `rate(http_requests_total[30s])` — берёт разницу между соседними значениями счётчика при scrape и делит на интервал.

**Состояние пула** снимается фоновой горутиной-коллектором каждую секунду через `pool.Stat().AcquiredConns()` и записывается в `Gauge`. Снимать это значение в middleware после завершения запроса бессмысленно — к тому моменту соединение уже возвращено в пул и `AcquiredConns` показывает 0. Только независимый тикер ловит состояние пула в момент пиковой нагрузки.

В Grafana четыре панели которые рассказывают одну историю. Первая — `rate(http_requests_total[30s])` с разбивкой по `status`: зелёный ряд success и красный ряд timeout. Вторая — `pgxpool_read_acquired_conns`: в эксперименте 1 упирается в `MaxConns` и застывает там. Третья — `pgxpool_write_acquired_conns`: остаётся около нуля потому что в этом эксперименте нет операций записи — это подтверждает что проблема именно в read-нагрузке. Четвёртая — `histogram_quantile(0.99, rate(http_request_duration_seconds_bucket[30s]))`: p99 latency растёт пропорционально глубине очереди — именно это и предсказывает закон Литтла.

## Эксперимент 1 — Воспроизводим connection exhaustion

**Цель:** при фиксированных `MaxConns = 25` и `query_duration = 100ms` найти уровень concurrency при котором `required_connections > MaxConns`, очередь переполняется и запросы начинают падать по таймауту.

**Расчёт до запуска.** При `concurrency = 50`: `required_connections = 50 × 0.1 = 5` — норма. Но пропускная способность пула `25 / 0.1 = 250 req/s`, а среднее время ожидания для горутины в хвосте очереди `(50 / 25) × 0.1 = 0.2s = 200ms`. При `context timeout = 150ms` горутины в хвосте очереди гарантированно получат таймаут.

```bash
docker compose -f compose.single.yml up -d
```

```bash
# Grafana: http://localhost:3000
# Prometheus: http://localhost:9090
ab -t 60 -c 50 http://localhost:8080/
```

**Что вы увидите.** `pgxpool_read_acquired_conns` упирается в 25 и не снижается. На графике RPS появляется красный ряд `timeout`. p99 latency на четвёртой панели растёт и стабилизируется на уровне значительно выше `query_duration` — это и есть цена очереди.

**Серия A — фиксируем `query_duration` и `context timeout`, меняем `concurrency`.** Запускайте последовательно с `-c 10`, `-c 25`, `-c 50`, `-c 100` и наблюдайте как меняется соотношение success/timeout. Точка перехода — когда `concurrency × query_duration` превышает `MaxConns × context_timeout`.

**Серия B — фиксируем `concurrency` и `context timeout`, меняем `query_duration`.** Замените `pg_sleep(0.1)` на `pg_sleep(0.05)` и повторите. Пороговый уровень нагрузки сдвинется ровно в два раза — это прямое подтверждение закона Литтла.

## Эксперимент 2 — Решаем через streaming replication

**Цель:** при тех же параметрах нагрузки добавить реплики и показать что `required_connections` распределяется по репликам, ни одна из них не достигает `MaxConns`, и таймауты исчезают — без единого изменения в коде приложения.

**Математика решения.** При 2 репликах эффективный пул ReadPool становится `25 × 2 = 50 соединений`. Среднее время ожидания при `concurrency = 50`: `(50 / 50) × 0.1 = 0.1s = 100ms` — укладывается в `context timeout = 150ms`. Все запросы успевают.

```bash
docker compose -f compose.ha.yml up -d
```

```bash
docker compose -f compose.ha.yml logs replica_1 replica_2
# ждём: "started streaming WAL from primary"
```

```bash
ab -t 60 -c 50 http://localhost:8080/
```

**Что вы увидите.** Те же четыре графика Grafana. `pgxpool_read_acquired_conns` не достигает потолка — нагрузка распределилась. Красный ряд `timeout` на нуле. p99 latency вернулась к `query_duration`.

**Горизонтальное масштабирование в действии.** Добавление третьей реплики требует только изменения конфига HAProxy. `MaxConns` в коде приложения остаётся прежним — `25`. Просто теперь за одним read-эндпоинтом стоят три независимых пула.

## Выводы

Проблема connection exhaustion определяется одной формулой: `concurrency × query_duration > MaxConns`. Это не проблема плохого кода — это математическое свойство системы с фиксированным пулом соединений. Понимание этой формулы позволяет предсказать поведение системы до эксперимента и проверить предсказание на живых графиках.

Вертикальное масштабирование через увеличение `MaxConns` имеет жёсткий предел: PostgreSQL создаёт отдельный OS-процесс на каждое соединение, и при сотнях соединений overhead по памяти становится неприемлемым ещё до того как упрётесь в `max_connections`.

Горизонтальное масштабирование через streaming replication сдвигает порог насыщения пропорционально числу реплик: `r` реплик дают `MaxConns × r` эффективных соединений для читающего трафика. Приложение при этом не знает о топологии — CQRS-разделение на уровне пулов и HAProxy как прозрачный балансировщик делают это масштабирование невидимым для кода.

Replication lag — метрика для мониторинга, не проблема для кода. Если lag растёт — алерт в ops-канал. Приложение остаётся stateless относительно топологии репликации.
